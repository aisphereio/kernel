# Gateway runtime and proto-first API generation

This document records the current Kernel direction for external traffic and
internal service calls.

## Decision

External browser/API traffic should enter a Gateway/BFF service first. Business
services should be gRPC-first internally. Direct HTTP generated by
`protoc-gen-go-http` remains available for compatibility, debugging, and small
internal services, but it is no longer the preferred public edge path.

```text
Client / Portal
  -> HTTP/JSON
  -> aisphere-gateway
     -> restx standard chain
     -> session/csrf/authn/authz/audit filters or BFF logic
     -> grpc-gateway generated handler OR hand-written BFF logic
  -> gRPC
  -> skill/agent/workflow services
```

## Tooling

`make tools` installs pinned local generators into `.bin`:

```text
protoc-gen-go
protoc-gen-go-grpc
protoc-gen-go-http
protoc-gen-go-errors
protoc-gen-grpc-gateway
protoc-gen-openapiv2
buf
```

`make api` runs `buf generate` and produces, from the same proto contract:

```text
*.pb.go          protobuf messages
*_grpc.pb.go     gRPC client/server
*_http.pb.go     Kernel direct HTTP transport
*.gw.pb.go       grpc-gateway HTTP/JSON -> gRPC bridge
docs/openapi/*   merged OpenAPI v2 output
```

Keep `google.api.http` annotations explicit. Do not rely on unbound-method
fallback for public APIs unless a service has an intentional internal-only path.

## Runtime chain

`restx.DefaultMiddlewares()` now mirrors the go-zero-style default chain while
staying on top of Kernel `transportx/http`:

```text
Trace
RequestContext injection
AccessLog
Metrics
SanitizeHeaders
SecurityHeaders
MaxConns
Breaker
Shedding
Timeout
Recover
MaxBytes
Gunzip
```

Optional browser edge filters:

```text
CORS
CSRF
SessionMiddleware
InternalJWT verifier/injector
```

`RequestContext` injects these values into `contextx`:

```text
request_id
trace_id
logx.Logger
metricsx.Manager
contextx.ServiceContext
```

That means handler/logic code can use one stable context surface instead of
passing request IDs, loggers, metrics managers, and clients through every method.

## ServiceContext pattern

`contextx.ServiceContext` re-exports `servicecontextx.Context`. Startup owns the
wiring:

```go
svc := contextx.MustNewServiceContext()
contextx.MustPutDependency(svc, skillClient)
contextx.MustPutDependency(svc, authzGuard)
svc.MustPutClient(servicecontextx.ClientRef{
    Name: "skill",
    Kind: servicecontextx.ClientKindGRPC,
    Target: "skill-service:9000",
    Client: skillClient,
})
```

Gateway/logic code receives a `context.Context` and fetches dependencies only
from the injected service context when necessary:

```go
skill := contextx.MustGetDependencyFromContext[skillv1.SkillServiceClient](ctx)
```

Generated code should prefer explicit struct fields when available, but this
context bridge gives workers, middleware, and generated gateway handlers a
consistent fallback.

## grpc-gateway mounting

Use `grpcgatewayx.NewHandlerFromEndpoint` for normal REST/JSON-to-gRPC bridging:

```go
h, err := grpcgatewayx.NewHandlerFromEndpoint(ctx, grpcgatewayx.HandlerConfig{
    Endpoint: "skill-service:9000",
    DialOptions: []grpc.DialOption{
        grpc.WithTransportCredentials(insecure.NewCredentials()),
        grpc.WithUnaryInterceptor(gatewayx.InternalJWTUnaryClientInterceptor(jwt, "skill-service")),
        grpc.WithStreamInterceptor(gatewayx.InternalJWTStreamClientInterceptor(jwt, "skill-service")),
    },
}, skillv1.RegisterSkillServiceHandlerFromEndpoint)
```

`grpcgatewayx.DefaultMuxOptions` propagates request metadata such as
`x-request-id`, `x-trace-id`, `x-authz-decision-id`, and `authorization` into the
outgoing gRPC metadata. Business authz remains explicit in Gateway/BFF logic or
resource-specific filters.

## Internal JWT

Gateway-to-service calls should not forward browser cookies. Use short-lived
internal JWTs or mTLS. Kernel provides:

```text
gatewayx.InternalJWTUnaryClientInterceptor
gatewayx.InternalJWTStreamClientInterceptor
gatewayx.InternalJWTUnaryServerInterceptor
gatewayx.InternalJWTStreamServerInterceptor
gatewayx.InternalJWTHTTPMiddleware
```

The backend service verifies the token and receives the actor in `authn` context.

## Boundary

`grpc-gateway` solves only this bridge:

```text
HTTP/JSON client -> Gateway -> gRPC backend
```

It does not solve session, CSRF, authz, audit, capability APIs, OPA, or BFF page
aggregation. Those stay in Gateway/IAM/Portal code on top of Kernel runtime.
